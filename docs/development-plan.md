# 星火 Spark 开发计划

> 制定日期：2026-07-19。基于当日项目现状分析制定，对齐《星火 Spark 产品设计文档 V2》第八部分"落地实施路线图"。
> 本文档随迭代滚动更新：完成项移入"现状概览"，与设计文档产生偏差时需在此注明原因。

## 阶段一北极星

**一个真实的业委会，用星火完成一次真实投票，结果被街道办接受。**

一切排期以此为最高优先级倒推。与此无关的工作（弱组织域、生态开放、多场景复制）在达成前不投入。

## 现状概览

设计文档当前处于 **阶段一（MVP 底座与单场景验证，0-6 个月）** 早期。

已完成：

- PC 桌面客户端骨架（Electron 26 + Vue 3 + Element Plus），主进程内核 + 渲染进程经 IPC 访问
- 域隔离安全模型：窗口创建时绑定可信域，IPC handler 经 `getCallerDomain` 强制隔离（`src/main/index.ts`、`src/main/domain-registry.ts`）
- LevelDB 三域隔离存储（系统域 / 插件域 / 存证域）+ SHA-256 链式存证（`src/main/db/`）
- 根身份管理：bip39 助记词派生 Ed25519，密码加密本地存储（`src/main/identity/root-id.ts`）
- 强组织基础：组织 CRUD、成员管理、P2P 推/拉同步（`src/main/organization/`、`src/main/p2p/`）
- 插件系统基础：域隔离 SDK（db / docs / runtime / p2p / evidence）、插件市场（Ed25519 签名信任链）、weibo-core 示例插件
- 主程序热更新：manifest 签名校验 + 反回滚 + macOS spctl 校验（`src/main/updater/`）
- 内核模块测试（16 个测试文件）与三平台 CI Release 流水线

进行中：

- 插件 SDK 浏览器端改造（`src/renderer/plugin-sdk-browser.ts` 有未提交改动及新增测试）

### 3. 数据同步策略体系

治理数据不能用单一 LWW 冲突处理（分区时会静默丢写）。底层提供多种方案，默认给最安全的，其余要求插件显式声明。

- [ ]  底层先支持两种冲突处理原语：**append-only 存证日志（默认）** 与 LWW（声明可选）
- [ ]  SDK collection schema 增加必填 `syncStrategy` 声明，在类型层面强制显式选择
- [ ]  治理类数据（投票、成员、账目）强制 append-only + 链式存证
- [ ]  其余原语（OR-set 等）待成员类数据同步暴露真实问题后再加，不预先实现

### 4. P2P 零运维工程

P2P 的核心目标是"工具归业主所有"：家庭闲置台式机即中心节点，业主零运维感知。所有 P2P 工作以"业主感觉不到它的存在"为验收标准。

- [ ]  NAT 穿透；直连失败时回退到不可信中继转发（noise 端到端加密，中继运营方无法读数据，不破坏信任模型）
- [ ]  组织加入路径以邀请人为引导节点（邀请链接携带邀请人节点地址），将公共引导设施需求降到最低
- [ ]  组织域数据 K 副本策略；UI 呈现"已同步到 m/n 节点"状态，让"我的投票记上没有"可感知
- [ ]  火种节点休眠处理 / 保活策略
- [ ]  数据自动管理：本地自动备份、存储配额与过期清理

### 5. 身份与存证

- [ ]  助记词备份改为二维码（用户截图 / 保存图片即可）；二维码 payload 用用户密码加密，使图片单独泄露不足以致险，并提示勿网络传输、可打印留存

## 待讨论（已记录，暂不排期）

- **社交恢复（m-of-n）**：利用业委会天然信任网络做身份恢复，解决"手机丢失 = 二维码丢失 = 身份丢失"。不紧急，待投票插件上线后评估。
- **最小公共基础设施的供给**：跨网段首次发现与中继回退仍需要少量公共组件（引导节点 / 中继），非盈利模式下由谁运行、以何种形式（志愿者、社区共建）待议。原则：公共组件必须不可信化、可替换。
